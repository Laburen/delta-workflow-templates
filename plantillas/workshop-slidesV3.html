// el problema

Sin método, el trabajo se desorganiza.

// objetivo del workshop
Que cualquier Delta adopte un método de planificación reproducible: analizar el relevamiento, crear HUs, partir en tareas atómicas, armar el backlog.
// jerarquía documental

Este workshop enseña el método. El stack Laburen y los procesos del equipo se documentan oficialmente en la guía web: laburen.gitbook.io/laburen-docs/7NYbsch7fTyhsarLow8z — fuente canónica.

En el repo de cada proyecto vive un resumen ejecutivo en CONTEXTO-EMPRESA-METODOLOGIA.md — pedido al agente para que el agente y el equipo tengan el contexto a mano, sin reemplazar la guía web.

// principio
Documentar con criterio.
Partir en tareas atómicas.
El Delta decide siempre.
La IA acelera, no reemplaza.
// nota del autor

Este workshop te da el proceso completo y disciplinado. Con experiencia vas a ajustar — saltar pasos, fusionar etapas, romper alguna regla con conciencia. Pero arrancá disciplinado: el método se rompe bien recién cuando lo seguiste sin romper.

Dos naturalezas de trabajo.

El workshop separa dos fases con necesidades distintas. Usá un modelo para cada una — la elección concreta es tuya.

Sugerencia: Claude Sonnet o un modelo equivalente rápido. A elección del Delta.

// setup · antes de arrancar

El repo es el contexto.

Todo lo que el agente necesita leer vive en el repo. No hay "Project en claude.ai", no hay adjuntos sueltos. Los prompts referencian rutas, no pegan contenido.

// estructura inicial

{proyecto}/
├── README.md
└── docs/
    ├── relevamiento-{proyecto}.{html,pdf,md}
    ├── CONTEXTO-EMPRESA-METODOLOGIA.md
    └── (resto se genera en las próximas etapas)

// ventajas

  • → Versionado del contexto con el código
  • → Continuidad entre sesiones
  • → Onboarding inmediato
  • → Tokens optimizados — lee solo lo que necesita
  • → Agnóstico de entorno y agente
01
// etapa uno
Análisis del relevamiento.

// qué produce

Un único documento docs/RESUMEN-RELEVAMIENTO.md en dos partes:

PARTE 1 · RESUMEN EJECUTIVO

Actores, flujos macro, modelo de datos, integraciones, reglas de negocio, pendientes del cliente, fuera de scope.

PARTE 2 · ANÁLISIS CRÍTICO

A. Bloqueantes duros — B. Decisiones arquitectónicas — C. Riesgos a mitigar — D. Consideraciones futuras.

// qué pelear del output

  • ✗
    Confundir pendiente del cliente con limitación técnica
    "Faltan credenciales" no es limitación técnica.
  • ✗
    Inflar el número de bloqueantes
    A veces son riesgos, no bloqueantes.
  • ✗
    Saltarse riesgos sutiles
    Concurrencia, idempotencia, OAuth refresh, rate limits.
02
// etapa dos
Decisiones arquitectónicas.

Para cada decisión arquitectónica genuina que identifiques en el relevamiento, un ADR.

✓ AMERITA ADR
  • → Difícil de revertir
  • → Afecta múltiples componentes o flujos
  • → 2+ opciones reales con trade-offs
  • → Cliente o equipo necesita conocer y aprobar

// estructura del ADR

Estado: abierto / DECIDIDO (fecha)
Sprint que afecta: N
Quién decide: Delta / cliente / mixto
Contexto: el problema, qué pasa si no se decide
Opciones: 2-3 reales — no inventar relleno
Trade-offs: tabla con criterios discriminantes
Recomendación: 2-3 párrafos
Qué necesito para cerrar
Impacto de no decidir
03
// etapa tres
Backlog · HUs primero.

// formato

### HUI-NNN: Título de negocio

Como [actor concreto — nombre, rol]
Quiero [acción atómica]
Para [beneficio de negocio]

#### Criterios de Aceptación:

1. Dado que [precondición]
   Cuando [evento]
   Entonces [resultado testeable]

[3 a 7 criterios — camino feliz + edge cases + errores]

// qué es una hu

Una Historia de Usuario describe una necesidad del negocio desde la perspectiva de un actor concreto.

No es una tarea técnica — es qué quiere lograr alguien y por qué. Las HUs son el punto de partida del que se derivan las tareas técnicas.

04
// etapa cuatro · ★ crítica
Tareas técnicas atómicas.

Las tareas técnicas se derivan de las HUs. Le pedimos al agente que las genere atómicas y en orden recomendado de desarrollo — no alfabético, no por HU: orden real considerando dependencias técnicas y ruta crítica.

// por qué atómicas

  • →
    Control · lotes chicos se revisan mejor, los errores no se acumulan
  • →
    Recuperación rápida · si una tarea falla, perdés horas, no días
  • →
    Estimación realista · lotes chicos se estiman con menos error
  • →
    PR review eficaz · 400 líneas se revisa bien, 1000 se aprueba sin leer
  • →
    (Bonus IA) evita que el agente comprima contexto a mitad de tarea

// criterios prácticos *

  • →
    Toca 1-3 archivos, máximo 5
  • →
    Estimación humana base ≤ 4 horas
  • →
    Sub-tareas concretas, accionables, secuenciales
  • →
    Criterios de aceptación testeables y locales
  • →
    Cada tarea cubre al menos una HU explícita

* los números concretos son heurísticas operativas, no estándares de la industria. Calibrá según tu equipo y stack.

// regla operativa
Si una tarea no entra en estos criterios, son varias tareas, no una.
// ejemplo real · t-if-010

El formato completo.

### T-IF-010: Templates WhatsApp — Redacción y envío a aprobación de Meta

**Prioridad:** CRITICA (RUTA CRITICA)
**Estimación:** 1-2hs (+ 24-48hs de espera por Meta)
**Dependencias:** Acceso al Meta Business Portfolio (Gabriel)
**Estado:** Pendiente — iniciar el día 1 del sprint
**Tarea Asana:** Sprint 1 — Circuito A
**Cubre HUI:** HUI-004

#### Descripción
Redactar el template consulta_disponibilidad requerido para el contacto
proactivo del Sprint 1 y enviarlo a aprobación de Meta. Sin este template
aprobado, HUI-004 no puede ejecutarse.

#### Tareas específicas
- [ ] Verificar templates ya existentes en la cuenta WA Business
- [ ] Redactar consulta_disponibilidad (Utility) con variables {{1..5}}
- [ ] Enviar el template a aprobación de Meta
- [ ] Monitorear el estado y re-enviar si Meta lo rechaza

#### Criterios de aceptación

// definición y carga

Del backlog al sprint.

Una vez generadas las tareas técnicas, se cargan todas en Asana como backlog operativo del proyecto. Desde ahí se arman los sprints. El docs/BACKLOG.md del repo sigue siendo la fuente técnica que el agente lee al ejecutar cada tarea.

// el flujo

  • 1
    Tareas generadas · el agente las escribe en docs/BACKLOG.md
  • 2
    Carga a Asana · las mismas tareas se trasladan al backlog operativo
  • 3
    Sprint planning · del backlog en Asana se arma el sprint en curso
  • 4
    Ejecución · el Delta toma las tareas del sprint; el agente lee el detalle desde BACKLOG.md

// dos fuentes complementarias

BACKLOG.md (repo)

Fuente técnica · detalle completo de cada tarea · lo que lee el agente al arrancar

ASANA

Fuente operacional · estado, asignación, sprint · lo que ve el equipo en el día a día

Las dos fuentes viven en paralelo durante todo el proyecto. El "cómo" operativo (quién carga, en qué momento) se define a nivel equipo.

// etapa cinco
Desarrollo + testing.

A partir de acá entrás en el ciclo que se repite por cada tarea T-XXX. Una tarea = una branch = un PR. El método de planificación ya terminó — esto es ejecución estándar.

// principios universales

  • →
    Tomar tarea asignada del sprint actual en Asana
  • →
    Feature branch desde develop · trigger phrase corta al agente
  • →
    TDD Red → Green → Refactor
  • →
    Testing manual E2E si la tarea es testeable (lo hacés vos)
  • →
    Quality gates antes del PR · CI verde
  • →
    Conventional Commits con T-XXX-NNN en el scope

// trigger phrase al agente

> Iniciar T-IF-010 del backend.

El agente lee AGENTS.md + workflow correspondiente + la tarea específica del backlog. No hace falta pegar contexto: el repo es el contexto.

// regla dura del delta
Si la tarea es testeable E2E, la probás vos antes del PR. No alcanza con tests automatizados verdes.
06
// etapa seis · ★ crítica
PR + feedback.

El reviewer de IA es preciso si está bien configurado. Lo que nunca hacemos es aplicar su feedback automáticamente: pasa por un protocolo de validación que termina con tu permiso explícito. Incluye el merge del Delta y el cierre en Asana.

// flujo agnóstico de reviewer

  • 1
    Configurar al reviewer con contexto del proyecto (.github/instructions/, .coderabbit.yaml, etc.)
  • 2
    Recibir feedback por el canal del reviewer
  • 3
    Pegar el feedback al agente de código
  • 4
    Agente categoriza cada item contra AGENTS.md
  • 5
    Reporta al Delta con formato estructurado
  • 6
    Aplica sólo tras permiso explícito · commits separados

// las 3 categorías

✓ VÁLIDO

Aplica las convenciones del proyecto · suma valor.

✗ INVÁLIDO

Contradice un ADR cerrado · sugiere any / eslint-disable · fuera de contexto.

⚠ DUDOSO

Preferencia estilística no cubierta · necesita decisión del Delta.

// adaptaciones posibles

CodeRabbit · feedback como comentarios en el PR + .coderabbit.yaml  ·  Qodo Merge · revisión automática on-push  ·  Cursor BugBot · integrado en el IDE  ·  Manual con Opus · si no hay reviewer configurado.

El canal cambia. El protocolo es el mismo: validar → permiso explícito → aplicar → re-verificar.

// regla central · cierre de la etapa 06
El merge
lo hace siempre
el Delta.

Nunca el agente, aunque diga "todo verde, mergeo". Antes de mergear, confirmás: CI verde · review IA procesado · testing manual hecho · sin conflictos. Si todo está, squash and merge y borrás la branch. Después marcás la tarea como completada en Asana.

// human-in-the-loop · cierre del workflow
// anti-patrones

Las seis trampas.

01 · planificación
Aceptar el primer output
Estás introduciendo errores que se propagan a todas las etapas siguientes.
02 · planificación
Tareas técnicas grandes
"Para que el agente las haga todas juntas." Garantía de auto-compact + bugs.
03 · desarrollo
Pasarle todo el AGENTS.md
El agente lo lee solo cuando lo necesita. Vos no lo cargás como prompt.
04 · desarrollo
Mergear sin testing manual
Los tests automatizados no cubren todo. El Delta prueba.
05 · review
Aplicar todo el feedback
Los IA reviewers se equivocan. Cada item se valida primero.
06 · review
Dejar que el agente mergee
Siempre el Delta. Sin excepciones, ni siquiera en hotfix.

// lo no negociable

Si te olvidás de todo...

...quedate con estas siete reglas:

01 Una tarea = una branch = un PR. Squash and merge a develop.
02 TDD siempre. Test primero, código después. Sin excepciones.
03 Sin any, sin @ts-ignore, sin --no-verify. RULE 0 tolerancia cero.
04 Tareas atómicas. ≤ 5 archivos, ≤ 4h humanas, ≤ 10 sub-tareas.
05 Testing manual E2E antes del PR. Si es testeable, lo probás vos.
06 PR feedback se valida antes de aplicar. Categorías ✓/✗/⚠ + confirmación explícita.
07 El merge lo hace siempre el Delta. Nunca el agente. Sin excepciones.
